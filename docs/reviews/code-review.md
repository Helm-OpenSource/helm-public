# 通用代码评审标准

## 目的

本文件用于统一“经营分身控制台”项目的代码评审标准。

评审的核心目标有四个：

1. 保证产品方向没有偏移
2. 保证核心流程没有被破坏
3. 保证代码质量与可扩展性
4. 保证演示路径始终稳定可用

本项目的评审标准，优先关注“用户是否更容易理解、信任、使用和购买这个产品”，其次才是纯技术层面的优雅程度。

---

## 一、评审总原则

每次代码评审，都先回答下面五个问题：

1. 这次改动是否让产品更好用？
2. 这次改动是否让核心页面更清晰？
3. 这次改动是否让审批、行动、机会流转更可信？
4. 这次改动是否会破坏现有演示路径？
5. 这次改动是否增加了不必要的复杂度？

如果以上五个问题里，有两个及以上回答为“否”或“不确定”，应要求补充修改。

---

## 二、产品方向评审

### 2.1 必须保持的产品定位

“经营分身控制台”是一个面向高价值职业个体和小团队的经营工作台。

核心价值始终围绕以下几点：

1. 帮用户识别今天最重要的经营动作
2. 帮用户把会议和沟通转成下一步动作
3. 帮用户减少机会漏损
4. 让 AI 在明确授权下执行部分动作
5. 让所有 AI 建议和动作都可审批、可追踪、可回放

### 2.2 方向偏移的红线

如果改动出现以下情况，应视为方向偏移：

1. 页面逐渐变成聊天工具主导
2. 功能越来越像通用 CRM
3. 功能越来越像邮件客户端
4. 功能越来越像知识库或笔记产品
5. 加入了和首期价值无关的社交、支付、数字人等能力
6. 出现大量“看起来很炫但对经营无帮助”的功能

### 2.3 评审问题

1. 这次改动是否强化了“经营分身”的定位？
2. 这次改动是否让用户更容易推进工作？
3. 这次改动是否增强了“可控行动代理”的感知？
4. 这次改动是否服务于首期四类目标用户？

---

## 三、核心流程评审

### 3.1 核心流程清单

评审时优先检查以下流程是否更顺、更清晰：

1. 今日工作台查看今日重点
2. 从会议进入 briefing
3. 从会议生成 action items
4. 从 action items 进入审批
5. 审批后回写机会、联系人、时间线、审计日志
6. 在机会面板推进状态
7. 在联系人页生成下一步动作
8. 修改策略后影响审批逻辑

### 3.2 关键判断标准

1. 用户是否知道下一步该做什么
2. 用户是否知道 AI 为什么建议这个动作
3. 用户是否知道哪些动作需要自己批准
4. 用户是否能看到批准之后发生了什么变化
5. 用户是否能追溯某个动作从哪里来

### 3.3 流程退化警报

如果出现以下任一情况，应要求修改：

1. 用户完成关键任务需要额外解释
2. 页面之间跳转过多
3. 一个动作做完后没有明确反馈
4. 数据已经变化，但用户不容易看出来
5. 策略修改后行为没有变化
6. 审批通过后相关对象没有联动更新

---

## 四、页面评审标准

### 4.1 今日工作台

评审重点：

1. 是否能让用户 5 秒内知道今天该做什么
2. 今日重点、待审批、高风险、今日会议是否足够醒目
3. 每个模块是否有明确 CTA
4. 信息密度是否合适
5. 是否有明显空状态和完成状态

### 4.2 机会面板

评审重点：

1. 看板是否足够像业务推进系统
2. 卡片信息是否足够支持判断
3. 拖拽状态变更是否有即时反馈
4. 机会详情抽屉是否能直接推进事情
5. 是否能识别高风险与逾期未推进机会

### 4.3 会议详情页

评审重点：

1. 会前 briefing 是否像真实简报
2. 会后动作是否足够清晰和可编辑
3. 一键生成纪要、发起审批、更新机会状态是否形成闭环
4. 页面是否足够支撑对外演示
5. 历史操作是否可追溯

### 4.4 审批中心

评审重点：

1. 是否建立信任感
2. 高风险与低风险动作区分是否明显
3. 批准、拒绝、编辑后批准三种路径是否顺手
4. 同类动作自动处理规则是否可理解
5. 审批后是否可见到明显结果

### 4.5 联系人页与公司页

评审重点：

1. 是否像动态工作档案
2. 是否能快速看懂关系现状
3. 是否能快速发起下一步动作
4. 是否与机会、会议、时间线真实联动

### 4.6 收件箱页

评审重点：

1. 是否像行动入口
2. AI 建议是否可操作
3. 升级为机会是否清晰
4. 线程与联系人、公司、机会之间关系是否明确

### 4.7 记忆与时间线页

评审重点：

1. 是否真的体现长期价值
2. 搜索和筛选是否有用
3. 记忆纠错和删除是否可信
4. 内容是否来自真实业务对象

### 4.8 设置与策略页

评审重点：

1. 是否容易理解
2. 每条策略是否有业务说明
3. 修改之后是否有可见影响
4. 是否避免沦为摆设页面

## 五、UI 与交互评审标准

### 5.1 统一性

检查以下内容是否统一：

1. 页面标题区
2. 卡片样式
3. 按钮层级
4. 表格样式
5. 抽屉和弹层
6. 筛选器
7. 标签和状态色
8. loading、empty、error 状态

### 5.2 信息层级

每个页面都应回答：

1. 最重要的信息在哪里
2. 风险信息在哪里
3. 下一步动作在哪里
4. 状态变化是否明显
5. 用户是否需要过多扫描才能理解页面

### 5.3 交互反馈

以下动作都必须检查反馈是否充分：

1. 保存
2. 审批
3. 拒绝
4. 拖拽变更状态
5. 新建对象
6. 编辑对象
7. 删除或归档
8. 修改策略

### 5.4 细节检查清单

1. 日期格式是否统一
2. 文案是否全中文且自然
3. 长文本是否做截断或 tooltip
4. 空状态是否有引导
5. 危险操作是否二次确认
6. Toast 是否清晰
7. CTA 是否可理解
8. 滚动和弹层是否顺滑

---

## 六、数据与业务逻辑评审

### 6.1 数据一致性

重点检查：

1. 同一个机会在不同页面的状态是否一致
2. 联系人、公司、会议、机会之间关系是否一致
3. 审批状态是否真实影响业务状态
4. 记忆条目是否来自真实对象
5. 时间线是否能解释关键变化

### 6.2 审批联动

每次涉及审批的改动，必须检查：

1. ApprovalTask 是否正确生成
2. 审批通过后目标对象是否更新
3. 审批拒绝后状态是否正确
4. AuditLog 是否记录
5. UI 是否反映新状态

### 6.3 策略联动

每次涉及策略修改，必须检查：

1. PolicyRule 是否更新
2. 相同类型动作的默认行为是否变化
3. UI 是否能看出策略生效
4. 审计日志是否记录策略变化

### 6.4 Demo 数据质量

每次改动都应确认 demo 数据仍然有故事性：

1. 销售故事线是否完整
2. 招聘故事线是否完整
3. 创始人故事线是否完整
4. 页面之间是否能串起来讲故事
5. 数据是否像真实业务数据

---

## 七、代码质量评审标准

### 7.1 代码结构

检查以下内容：

1. 是否遵循 feature-first 组织方式
2. 是否出现重复组件
3. 是否出现重复业务逻辑
4. 页面文件是否过于臃肿
5. 服务层是否清晰
6. 类型定义是否清楚

### 7.2 可维护性

1. 命名是否准确
2. 函数职责是否单一
3. 组件职责是否清晰
4. 是否存在难以理解的条件分支
5. 是否有明显坏味道代码
6. 是否有必要注释

### 7.3 类型与校验

1. 是否存在明显 any 滥用
2. 写操作是否做 Zod 校验
3. Server actions 或 route handlers 是否有输入约束
4. Prisma 类型是否被正确利用
5. UI 层是否假设数据一定存在

### 7.4 状态管理

1. 本地状态是否过多散落
2. 跨页面共享状态是否合理
3. 是否有不必要的复杂状态管理
4. 状态变化是否容易追踪

---

## 八、审计与可追责评审

以下行为必须确认 AuditLog 存在且可读：

1. AI 生成建议
2. 批准动作
3. 拒绝动作
4. 自动执行动作
5. 机会阶段变化
6. 策略变化
7. 联系人合并
8. 记忆修正
9. 记忆删除
10. 负责人变更
11. 截止日期变更

评审时必须检查：

1. 日志是否人类可读
2. 是否能关联 actor、target、summary、payload
3. 页面中是否能看见日志
4. 日志是否帮助解释“为什么这样变了”

---

## 九、演示路径评审

每次较大改动后，至少走一遍以下路径。

### 路径 A：Founder
1. 登录 founder 账号
2. 打开今日工作台
3. 查看今日重点
4. 进入会议详情
5. 查看 briefing
6. 生成会后动作
7. 发起审批
8. 批准动作
9. 查看机会变化与审计日志

### 路径 B：Sales lead
1. 打开机会面板
2. 拖动机会到新阶段
3. 进入联系人详情
4. 生成跟进草稿
5. 去审批中心批准
6. 回到联系人页和机会页验证变化

### 路径 C：Recruiter
1. 进入招聘机会
2. 查看候选人联系人页
3. 进入面试会议页
4. 生成后续动作
5. 审批安排
6. 查看时间线和审计日志变化

### 路径 D：策略变化
1. 修改外发消息策略
2. 重新触发动作
3. 验证审批逻辑变化
4. 验证日志记录

只要上述路径中任一断裂，就应视为回归问题。

---

## 十、性能与可靠性评审

检查以下内容：

1. 页面初次加载是否明显卡顿
2. 列表与抽屉切换是否顺畅
3. 是否存在明显闪烁
4. 是否存在无意义重复请求
5. 写操作失败是否有清晰提示
6. 空数据、脏数据、缺失数据是否有保护

---

## 十一、评审结论模板

每次评审建议按以下结构输出：

### 1. 总体结论
一句话总结当前改动是否可以合并。

### 2. 做得好的地方
列出 2 到 5 个值得保留的点。

### 3. 必须修改的问题
只写真正影响产品质量、流程闭环、可信度、演示稳定性的问题。

### 4. 建议优化的问题
写那些可以进一步提升成熟感和可卖性的点。

### 5. 验证情况
说明你走了哪些 demo 路径，做了哪些检查。

### 6. 风险提醒
指出哪些地方后续继续迭代时容易出问题。

---

## 十二、合并门槛

以下情况可以合并：

1. 核心路径无断裂
2. 页面可理解
3. 状态联动正确
4. 审计日志完整
5. 代码质量可接受
6. 无明显 UI 断裂
7. Demo 数据仍可讲完整故事

以下情况不应合并：

1. 产品方向出现偏移
2. 核心路径断裂
3. 审批或策略变成假功能
4. 审计日志缺失
5. Demo 数据故事被破坏
6. 关键页面体验明显退化
7. 增加了大体量复杂度但没有等价产品价值

---

## 十三、评审优先级顺序

当时间有限时，按下面顺序评审：

1. 今日工作台
2. 会议详情页
3. 审批中心
4. 机会面板
5. 数据联动与审计
6. 联系人和公司详情
7. 收件箱
8. 记忆页
9. 设置与策略
10. UI polish
11. 工程代码整洁度

---

## 十四、评审者提醒

请始终记住：

这个项目的第一目标，不是功能堆叠。
这个项目的第一目标，是让用户在高频经营动作里，第一次感受到“一个可控的行动代理真的能帮我推进工作”。

任何改动，只要让这个感受更强，就是加分。
任何改动，只要让这个感受变弱，就是减分。
