# 付费 PoC 模板

## 文档目的

本文件用于指导“经营推进控制台”首批付费 PoC 的设计、报价、范围控制与交付节奏。

目标有四个：

1. 帮团队快速拿到第一批真实付费客户
2. 避免 PoC 变成无限制定制项目
3. 让客户和团队都清楚成功标准
4. 为后续标准化产品和定价建立基础

本模板适用于：

1. 企业服务销售团队
2. 猎头和招聘顾问团队
3. 创始人 / COO 团队
4. 高客单价顾问团队

---

## 一、PoC 核心原则

### 1. PoC 不是“定制开发”
PoC 的目标是验证：
- 产品是否解决关键问题
- 客户是否愿意使用
- 客户是否愿意继续付费扩展

不要把 PoC 做成：
- 无限需求收集
- 无边界定制
- 没有明确成功标准的试点

### 2. 每个 PoC 只打一个核心结果
建议从下面四类结果里选一个：

1. 减少会后漏跟进
2. 提升机会推进可见性
3. 提升审批与动作可控性
4. 提升招聘 / 客户推进效率

不要一个 PoC 同时承诺解决所有问题。

### 3. 先验证高频场景
PoC 优先只覆盖：
- 会议
- 邮件 / 线程
- 联系人
- 机会
- action items
- 审批

### 4. 必须有共同定义的成功标准
没有成功标准的 PoC，基本都会失败。

---

## 二、PoC 标准结构

### 1. PoC 名称
例如：
- 销售团队会后推进 PoC
- 猎头团队职位推进 PoC
- 创始人经营控制台 PoC

### 2. PoC 周期
建议：
- 2 周版本：极轻量验证
- 4 周版本：标准试点
- 6 周版本：较深度共创

推荐标准版本为 4 周。

### 3. 参与角色
客户侧至少需要：

1. 业务负责人
2. 实际使用者 2 到 5 人
3. 数据 / IT 支持人
4. 项目对接人

你方至少需要：

1. 产品负责人
2. 交付 / 客户成功负责人
3. 技术支持负责人

### 4. 数据范围
PoC 只接入必要数据，不做全量系统改造。

推荐首批数据范围：
1. 联系人 CSV
2. 当前机会台账
3. 历史会议纪要
4. Gmail 只读
5. Calendar 只读

---

## 三、PoC 范围模板

### A. 销售团队 PoC

#### 目标
减少会后漏跟进，提升机会推进可见性。

#### 范围
1. 首页今日重点
2. 会议 briefing 与会后 action
3. 机会面板
4. 联系人页
5. 审批中心
6. 简单管理者周报

#### 不在范围内
1. 真实支付
2. CRM 全量替换
3. 所有客户历史数据重建
4. 所有流程自动化

#### 验证指标
1. 会后 action item 生成率
2. action item 被采纳率
3. 高风险机会识别数
4. 被批准的外发动作数
5. 使用频次

---

### B. 招聘团队 PoC

#### 目标
减少职位推进和候选人推进断点。

#### 范围
1. 职位机会面板
2. 候选人联系人页
3. 面试会议页
4. 跟进草稿生成
5. 审批与时间线
6. 每周推进摘要

#### 验证指标
1. follow-up 延迟减少
2. 候选人流失预警识别
3. 面试后 action item 采纳率
4. 顾问使用频次
5. 被批准的对外动作数

---

### C. 创始人 / COO PoC

#### 目标
让创始人 / COO 获得一个每日经营控制面。

#### 范围
1. 今日工作台
2. 关键会议 briefing
3. 内部事项和外部机会统一面板
4. 审批中心
5. 周报与高风险事项
6. 简单策略管理

#### 验证指标
1. 每日打开频次
2. 高风险事项识别数
3. 被批准动作数
4. 会议后推进完成率
5. 用户主观价值评分

---

## 四、PoC 成功标准模板

每个 PoC 必须至少定义以下 5 项成功标准。

1. 使用标准
   - 至少 X 名用户参与
   - 每周至少使用 Y 次

2. 数据接入标准
   - 至少接入联系人 / 机会 / 会议中的两类真实数据

3. 过程标准
   - 至少完成 Z 场真实会议到 action 的闭环

4. 结果标准
   - 至少在一个核心指标上有可感知改善
   - 例如减少漏跟进、提高 action item 采纳率

5. 合作标准
   - 客户愿意参与中期和结项复盘
   - 客户愿意给出是否继续扩展的明确结论

## 五、PoC 交付节奏模板

### 第 0 周：确认
1. 明确 PoC 目标
2. 明确范围和边界
3. 明确参与人
4. 明确成功标准
5. 明确付款方式

### 第 1 周：接入与初始化
1. 接入数据
2. 配置工作区
3. 完成初始演示
4. 培训核心用户

### 第 2 周：真实使用
1. 跑真实工作流
2. 记录问题
3. 做第一轮小修
4. 输出中期复盘

### 第 3 周：强化闭环
1. 聚焦一个最核心流程
2. 增强审批、时间线、推荐
3. 输出中期价值观察

### 第 4 周：结项与决策
1. 输出 PoC 结果报告
2. 对照成功标准打分
3. 给出是否进入下一阶段建议
4. 讨论扩展范围和正式合作模式

## 六、报价模板

建议首批 PoC 明确收费，不做纯免费。

### 方案 A：轻量验证版
- 周期：2 周
- 适合：1 到 3 个用户
- 价格建议：3 万到 5 万元
- 范围：单一场景验证
- 目标：快速判断是否值得继续

### 方案 B：标准试点版
- 周期：4 周
- 适合：3 到 10 个用户
- 价格建议：8 万到 15 万元
- 范围：完整小闭环
- 目标：验证使用和续费可能性

### 方案 C：共创验证版
- 周期：6 周
- 适合：5 到 20 个用户
- 价格建议：15 万到 30 万元
- 范围：更深工作流共创
- 目标：验证更高价值场景

---

## 七、PoC 结果报告模板

每个 PoC 结束后，必须形成一份结果报告，至少包含：

### 1. 背景
- 客户是谁
- 验证什么问题
- 周期多久
- 参与人数

### 2. 范围
- 接入了哪些数据
- 覆盖了哪些页面和流程
- 没做哪些事情

### 3. 使用情况
- 登录次数
- 会议数
- action item 生成数
- 审批数
- 用户活跃度

### 4. 结果观察
- 会后推进是否改善
- 漏跟进是否减少
- 审批体验是否建立信任
- 哪类页面最有价值

### 5. 用户反馈
- 最喜欢什么
- 最大担忧是什么
- 希望下一步解决什么

### 6. 结论
- 是否继续推进
- 是否扩展范围
- 是否进入正式合作

---

## 八、PoC 过程中的红线

以下情况必须及时止损或重新谈边界：

1. 客户不断追加无边界需求
2. 客户没有明确负责人
3. 客户不给真实数据也不给真实场景
4. 客户不参与复盘
5. 客户没有明确成功标准
6. 客户想把 PoC 当长期免费使用

---

## 九、PoC 成交时要明确的 8 件事

1. 核心目标是什么
2. 范围是什么
3. 不在范围内的是什么
4. 数据怎么给
5. 谁负责配合
6. 如何判断成功
7. 周期多久
8. 如何收费与付款

---

## 十、首批 PoC 的建议打法

建议首批只做 2 个 PoC：

1. 一个销售团队
2. 一个猎头团队

原因：
1. 场景高频
2. 结果直接
3. 页面价值容易体现
4. 审批和会后 action 的价值容易被感知

创始人 / COO 场景更适合做高层演示和第二波试点。

---

## 十一、PoC 成功的信号

如果出现以下信号，说明方向对了：

1. 客户开始主动每天打开系统
2. 客户开始把真实流程迁入
3. 客户愿意让 AI 自动做部分低风险动作
4. 客户开始问更多接入问题
5. 客户开始讨论扩展到更多人或更多场景
6. 客户愿意讨论正式采购或更长期合作

---

## 十二、最后提醒

PoC 的本质不是证明你能做很多功能。
PoC 的本质是证明：

**在一个真实高频场景里，这套产品能更稳定地推进事情，而且客户愿意继续用。**

只要这个结论成立，后面的功能扩展和产品深化才有价值。
