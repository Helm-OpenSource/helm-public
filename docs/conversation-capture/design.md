# Conversation Capture 设计稿

## 文档目的

本文件用于定义“经营分身控制台”的交流捕获系统（Conversation Capture System）。

目标有五个：

1. 让系统拥有一个高频、真实、现场发生的数据入口
2. 让产品更像经营分身，而不是事后整理工具
3. 让大量 AI 智能处理发生在后台，前台保持简洁易用
4. 让输出最终回到经营和管理体系，形成闭环
5. 让会议、电话、对话记录真正转化为推进动作、承诺链和风险识别

---

## 一、产品定位

交流捕获系统不是一个“录音工具”。

它是经营分身控制台的 **现场入口层**。

一句话定义：

**开始记录 → 自动转写 → 自动提取经营信号 → 自动形成记忆与动作 → 回到经营和管理闭环。**

关键在于：
- 用户只做一个动作：开始记录
- 后台自动完成大量 AI 处理
- 前台只呈现最值得看的结果和最值得做的动作

---

## 二、产品原则

### 1. 简洁
用户不应该被复杂设置淹没。
最理想的前台交互是：
1. 开始记录
2. 结束记录
3. 查看结果
4. 审批或推进

### 2. 后台智能处理
AI 处理应集中发生在后台，包括：
1. 语音转写
2. 说话人分离
3. 关键事实提取
4. 承诺提取
5. 阻碍识别
6. 风险识别
7. next best action 生成
8. briefing / recommendation 增强

### 3. 回到经营闭环
所有结果都要回到既有经营对象中，而不是成为独立的录音孤岛。

输出必须回写到：
- Contact
- Company
- Opportunity
- Meeting
- MemoryFact
- Commitment
- Blocker
- Recommendation
- ApprovalTask
- AuditLog

### 4. 可控
系统不能因为“听到更多”就越权。
任何高风险动作仍然必须回到：
- policy
- approval
- audit

### 5. 有现场感
产品不是让用户“事后上传录音”，而是让经营分身和用户一起在场。

---

## 三、为什么这一层重要

这层的价值不在录音本身，而在三点：

### 1. 它带来真实数据入口
很多关键经营信息并不出现在 CRM 字段里，而出现在对话里：
- 真正顾虑
- 口头承诺
- 隐性阻碍
- 关系温度变化
- 决策人态度

### 2. 它让记忆系统更真实
相比手工补录，现场捕获更接近真实意图，形成的：
- MemoryFacts
- Commitments
- Blockers
- Recommendation explanation
会更可信。

### 3. 它让产品更像分身
一旦用户养成“重要交流时先打开系统”的习惯，产品就不只是后台系统，而是现场参与者。

---

## 四、理想用户感受

用户应该感受到的不是：
“这里可以录音。”

而是：
1. 我开始一次重要交流时，这个系统就在场
2. 结束后，它自动把关键信息组织好了
3. 我不需要再手工复述发生了什么
4. 系统自动帮我盯住承诺、阻碍和下一步
5. 事情会被继续推动，而不是停在纪要里

---

## 五、前台产品形态

## 1. 主入口设计

主入口建议叫：

- 开始记录
- 开始捕获
- 进入会话模式

不建议直接叫：
- 录音
- 录音工具
- 上传录音

推荐做法：
1. 在首页放固定主入口
2. 在 Contact / Company / Opportunity / Meeting 页面提供“从这里开始记录”
3. 在连接器页提供 Zoom / 腾讯会议 / Call Center 接入配置

## 2. 前台流程设计

### 流程 A：手动现场记录
1. 用户点击“开始记录”
2. 选择上下文：
   - 联系人
   - 公司
   - 机会
   - 会议
   - 稍后归类
3. 系统进入录音中状态
4. 用户结束记录
5. 系统进入处理状态
6. 输出结果页：
   - 本次关键结论
   - Commitments
   - Blockers
   - Risks
   - Next best actions
   - 是否进入审批 / 是否自动创建 action items

### 流程 B：外部会议自动接入
1. 用户绑定 Zoom / 腾讯会议 / Call Center
2. 会议结束后自动进入处理链
3. 输出直接进入：
   - 会议页
   - 机会页
   - 联系人页
   - 首页
   - 审批中心

### 流程 C：现场轻量标注
在录音过程中支持简单标记：
- 重要
- 有承诺
- 有风险
- 要跟进

这个功能可放在后续阶段，不作为第一阶段必做。

---

## 六、系统核心输出

系统不要只输出全文转写。
最有价值的输出有五类：

### 1. 关键事实
例如：
- 客户更关心付款周期
- 候选人对薪资有顾虑
- 对方倾向先看轻量方案

### 2. 承诺
例如：
- 下周三前发方案
- 48 小时内同步反馈
- 本周内确认资源排期

### 3. 阻碍
例如：
- 预算审批慢
- 候选人对稳定性犹豫
- 内部资源冲突

### 4. 风险
例如：
- 若 24 小时内不 follow-up，关系会降温
- 当前 blocker 若不解决，机会可能继续停滞

### 5. 推进行动
例如：
- 发精简版结构草稿
- 安排下一次会议
- 内部确认资源排期
- 发反馈并进入审批

---

## 七、与现有系统的闭环关系

交流捕获层不是单独功能，必须回到现有闭环里。

### 1. 回到记忆系统
输出：
- MemoryFact
- Commitment
- Blocker
- Timeline

### 2. 回到 recommendation
输出：
- supporting facts
- new blockers
- new commitments
- updated context

### 3. 回到审批中心
输出：
- 外发草稿
- 高风险承诺动作
- 对外确认动作

### 4. 回到首页
输出：
- 今日高优先事项
- 逾期承诺
- 新识别风险
- 系统已推进事项

### 5. 回到管理者周报
输出：
- 高频 blocker
- 新增承诺
- 会议后的推进质量
- 风险分布变化

---

## 八、场景优先级

## P0 场景
优先做：
1. 手动开始记录
2. 自动转写
3. 自动提取 facts / commitments / blockers / risks / next actions
4. 写回会议页、联系人页、机会页、首页、审批中心

## P1 场景
再做：
1. Zoom / 腾讯会议接入
2. 说话人分离
3. 关键片段高亮
4. recommendation explanation 引用对话片段

## P2 场景
后续做：
1. Call Center API / CLI 接入
2. 会中实时标记
3. 实时风险提示
4. 更复杂实时 agent 协助

---

## 九、必须避免的方向

1. 做成独立录音工具
2. 做成事后上传工具
3. 做成只会生成长纪要的 AI 摘要器
4. 做成不回写经营对象的孤立能力
5. 做成无授权提示的隐形录音系统
6. 做成没有置信度、没有纠错入口的黑盒理解层

---

## 十、最终判断

交流捕获系统的本质不是“让产品多一个录音按钮”。

它的本质是：

**让经营分身真正进入现场，并把现场交流直接转化为经营记忆、推进动作和管理闭环。**

只有这样，它才会显著增强：
1. 智能感
2. 粘度
3. 依赖感
4. 数据资产密度
5. 经营分身感
