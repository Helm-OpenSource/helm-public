# Pack A 4 周 Pilot 执行手册 V1

## 1. 目的
为 Pack A 第一个 design partner 客户的 4 周 paid pilot 提供端到端执行指引。**这是第一次成果展示的种子事件**——按 Pack A V2 §11 强约束 C，本 runbook 必须保证客户首日感知良好，并把 4 周末作业质量验收拆成继续观察、续约谈判、公开 proof 三档，避免首个 pilot 因单一硬门过早失败。

**当前状态：** 本文件是 pilot 执行模板，不代表已经有客户合同、DPA、工作区、连接器、Pack A runtime 或 paid pilot 启动。所有 `完成` / `通过` 项均为启动前检查口径或实跑记录占位。

## 2. 适用客户与前置条件

### 2.1 适用
- Pack A 第一个 / 第二个 / 第三个 design partner 客户
- 通过 PACK_A_DESIGN_PARTNER_EVALUATION_V1 评分 ≥70 分
- paid pilot 已签署，默认报价锚点为 ¥50,000，但允许在 Top 2-3 scope call 中测试 `¥30k / ¥50k / ¥80k` 三档；不接受 0 元 PoC 作为第一个 design partner 启动条件

### 2.2 前置（启动前必须完成）
| 项 | 状态 | 责任方 |
|---|---|---|
| 客户合同 + 数据处理协议（DPA）签署 | 待完成 | Helm + 客户法务 |
| 工作区开通 + 初始管理员账号 | 待完成 | Helm 工程 |
| Pack A 默认配置（pack.config.yaml）部署 | 待完成 | Helm 工程 |
| 客户运营负责人指定（销售总监级） | 待完成 | 客户 |
| 4 周 review 节奏定档（每周 1 次） | 待完成 | 双方 |
| 创始人首次现场访谈日确定 | 待完成 | 客户 + 创始人 |
| 连接器接入工时评估完成（认证工程师对接） | 待完成 | 工程师 + 客户 IT |
| Helm 内部 founder / GTM operator / implementation lead 最小分工确认 | 待完成 | Helm |

**任何一项未完成**：暂缓启动，不强行进入 pilot。

### 2.3 启动前离线 gate

进入 Top 2-3 scope call 或 Week 0 前，先运行：

```bash
npm run eval:pack-a-pilot-readiness
```

该命令把 Pack A OPC 评分、Week 0 启动条件和 4 个 Pack A Skill 的集成覆盖统一成一个离线 readiness readout。它只使用 alias-only fixture，不接真实客户数据；通过不代表可以跳过合同、DPA、客户授权、法务 review 或 founder 最终判断。

## 5. Week 1：Day-1 印象（D1 ~ D7）

### 5.1 D1：客户首日体验（最关键的一天）

**上午（09:00-12:00）：**
- 09:00 连接器最终联通确认
- 09:30 Day-1 看板加载 fixtures（让首日有可读内容）
- 10:00 创始人 + Helm 工程现场陪伴客户运营负责人首次登录
- 10:30 销售 VP / 销售总监级别 demo（30 分钟）：
  - 讲清 4 个 Skill 价值
  - 讲清"建议 vs 承诺"边界
  - 讲清 4 周里程碑
- 11:00 销售代表 1:1 上手（每销售 15 分钟）
- 12:00 简短反馈采集（"今天感觉如何？"）

**下午（14:00-18:00）：**
- 14:00 真实会议数据接入（替换 Day-1 fixture）
- 15:00 第一次真实 A1 会议跟进清单生成（销售实际使用）
- 16:00 销售总监 / VP 第一次打开 A3 主管面板
- 17:00 客户运营负责人记录"首日感知"（结构化反馈表）

**Day-1 成功门：** 客户运营负责人主观评分 ≥3.5/5（5 分制：糟糕/一般/不错/好/非常好）。低于 3.5 立即进入 D2 紧急调优。

### 5.2 D2-D5：日常使用 + 首次反馈
**每日例行：**
- 销售使用 A1 + A2
- 销售总监使用 A3（仅主管可见）
- 已成单客户启动 A4 Handoff（如 D2-D5 期间有成单）
- Helm 工程师每日抽样 ≥3 条 audit chain，看推荐质量

**D5 反馈采集（结构化）：**
| 维度 | 采集方法 |
|---|---|
| A1 跟进清单采纳率（打开率 + 编辑率） | 数据自动采集 |
| A2 排序与销售心目中前 5 重合度 | 销售自报 + 主管对照 |
| A3 介入有效率 | 主管反馈 |
| 越界承诺识别准确性（FP/FN） | 销售总监抽样审核 5 条 |
| 整体可用度 | 销售个人 + 主管主观评分 |

### 5.3 D7：第 1 周 review（1 小时）
**议程：**
1. D1-D7 数据汇报（采纳率、识别率、Handoff 完整度）
2. 客户运营负责人主观评分
3. 销售代表口头反馈
4. **作业质量初判**：是否在 4 周 pilot 目标轨道上
5. Week 2 调优计划

**Week 1 通过门：** 4 项作业质量初步指标至少 50% 达到 4 周 pilot 目标的中位水平。

**Week 1 失败 / 中止信号（任一即告警）：**
- 客户运营负责人主观评分 < 3.0
- 销售代表 50%+ 反馈"不会用 / 不愿用"
- 连接器持续不稳定（≥3 次中断）
- 越界承诺识别 FP > 50%

告警触发立即进入 §10 中止流程评估。

## 7. Week 3：作业质量验证（D15 ~ D21）

### 7.1 量化数据采样
| Skill | 采样规则 |
|---|---|
| A1 | 全量统计跟进延迟、采纳率；销售总监每日抽 5 条审核越界承诺识别 |
| A2 | 销售主管每日对比"系统排序 vs 自己心目中"，记重合度 |
| A3 | 主管标记"介入有效 / 无效"；记录介入时机准确性 |
| A4 | CS lead 月度复盘 3 个 Handoff Pack；销售经理复盘完整度 |

### 7.2 业务侧主观评分（强约束 B 的关键）
| 评分人 | 频率 | 评分对象 |
|---|---|---|
| 销售总监 | 每周 1 次 | A1 + A2（5 条样本） |
| 销售 VP | 每周 1 次 | A3（自己介入命中率） |
| CS lead | 每 2 周 1 次 | A4（3 个 Handoff Pack） |

**评分量表（统一 5 分制）：**
- 1：完全不可用
- 2：要大改才能用
- 3：可用但需要常修
- 4：好用，偶尔需要修
- 5：直接可用

**Week 3 通过门：** 全部业务侧主观评分均值 ≥3.5。

### 7.3 D21：第 3 周 review
**议程：**
1. 量化数据 + 主观评分汇总
2. 是否达成 4 周 pilot 验收目标的初步判断
3. Week 4 验收日程定档

## 8. Week 4：验收与转化（D22 ~ D28）

### 8.1 D22-D27：作业质量验收
按 Pack A V2 §8 acceptance.pilot_4w 全量验收：

| Skill | 4 周目标 | 实测 | 通过 |
|---|---|---|---|
| A1 跟进延迟 p50 | ≤30 分 | _填_ | ✅/❌ |
| A1 采纳率 | ≥60% | _填_ | ✅/❌ |
| A1 越界承诺识别准 | ≥70% | _填_ | ✅/❌ |
| A2 排序重合度 | ≥60% | _填_ | ✅/❌ |
| A2 漏判率 | ≤20% | _填_ | ✅/❌ |
| A3 介入有效率 | ≥40% | _填_ | ✅/❌ |
| A3 介入时机准 | ≥50% | _填_ | ✅/❌ |
| A4 完整度 | ≥3.5/5 | _填_ | ✅/❌ |
| A4 CS 可读性 | ≥3.5/5 | _填_ | ✅/❌ |
| 业务侧主观评分均值 | ≥3.5/5 | _填_ | ✅/❌ |

**分层验收门：**

| 通过项 | 决策 | 含义 |
|---:|---|---|
| 0-3 / 10 | Stop / 回炉 | Pack A 当前形态不适合该客户；不进入续约，不沉淀公开 proof |
| 4-5 / 10 | 继续观察 | 痛点可能成立，但产品 / 数据 / review 节奏未达商业化；可延长 2-4 周或回到内部校准 |
| 6-7 / 10 | 续约谈判 | 可进入 paid continuation / quarterly observation，不允许写成公开成功案例 |
| ≥8 / 10 | 公开 proof 候选 | pilot 通过，可准备匿名 proof / 案例素材；仍需客户授权和 Helm 边界审计 |

`≥8 / 10` 不再是一票式“是否继续”的唯一门槛，只是公开 proof 和强续约叙事的门槛。

### 8.2 D28：4 周 review + 续约决策

## 10. 失败 / 中止流程

### 10.1 中止信号（任一即评估）
- Week 1 主观评分 < 3.0
- 任何 1 周连续 50%+ 销售反馈"不会用 / 不愿用"
- 连接器持续不稳定 7 天以上
- 客户运营负责人不参加 ≥2 次周 review
- 客户出现合规 / 法律纠纷影响 pilot

### 10.2 中止决策流程
1. Helm 创始人 + 客户运营负责人 1:1 评估
2. 决策：调整 / 暂停 / 中止
3. 中止情况下：
   - 退还未消耗 pilot 费用比例
   - 数据导出 + 销毁
   - 不留客户案例
   - 善意退出，保留二次接触可能

### 10.3 通过但未续约
- 4 周通过但客户决定不续约：
  - 不强求；保留客户作为案例线索
  - 6 个月后回访 1 次

## 11. 工程支持 SOP（pilot 期间）

### 11.1 7×12 工程响应
- 工作日 09:00-21:00 客户问题 ≤2 小时响应
- 周末紧急（连接器中断）≤4 小时响应
- 由认证工程师为主，Helm 工程团队为兜底

### 11.2 监控告警
- worker 失败率 > 5% → 工程立即介入
- 推荐 vs 承诺边界违反 → 实时告警 + 创始人通知
- 不掉案件 invariant 失败 → 阻断 + 告警

### 11.3 数据导出
客户随时可申请：
- audit chain 导出（90 天）
- 全部 Skill 输出导出
- 配置导出

## 12. pilot 期间的常见问题与处理

| 问题 | 处理 |
|---|---|
| "AI 推荐的跟进太死板" | A1 措辞模型温度调高；增加客户语料训练 |
| "我不喜欢主管知道我哪个会要被介入" | 引导销售用"私密会议"标记机制；解释 audit 边界 |
| "我们 CRM 里有些字段你没读到" | 认证工程师评估自定义字段映射（L2 工时） |
| "越界承诺识别太严格 / 太松" | 调整红线词典 + 通知阈值（Week 2 调优范围） |
| "Handoff Pack 太长，CS 看不下去" | 调整完整度评分公式 + 摘要分层（Week 2 调优） |

## 13. 风险与未决问题

1. **首日感知失败的连锁反应** — Day-1 印象差会导致后续 4 周难翻盘。**对策**：D1 上午 Helm 创始人现场陪伴，确保第一次体验顺
2. **客户运营负责人投入度递减** — 4 周内可能出现"前两周配合，后两周懈怠"。**对策**：每周 review 邀请客户 CEO/COO 列席（即使不发言）
3. **作业质量主观评分通胀** — 客户为了配合可能给高分。**对策**：邀请客户独立的销售代表（非负责人）独立打分对照
4. **认证工程师服务质量不齐** — 早期 L2/L3 工程师不足时，可能用初级。**对策**：第一个 design partner 必须由 Helm 内部人员或 L2 以上认证工程师交付
5. **数据接入比预估慢** — 自建 CRM 可能 3 周才能接通。**对策**：Week 0 评估时严格筛选，自建 CRM 的客户不作为第一个 design partner
6. **创始人被交付吞掉** — 首单启动后创始人同时承担销售、交付、访谈、续约和 proof，后续候选池中断。**对策**：kickoff 前确认 GTM operator 与 implementation lead；若创始人交付负载预计 ≥70%，第二个 design partner 不进入 kickoff
