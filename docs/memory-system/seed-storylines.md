# 记忆系统 Seed 故事线

## 文档目的

本文件用于为“经营分身控制台”的记忆系统提供高质量 seed 数据故事线规范。

目标有四个：

1. 让 Codex 在生成 seed 数据时有清晰标准
2. 让产品 demo 能讲出真实的经营故事，而不是零散数据
3. 让记忆系统、审批、审计、时间线、推荐链在演示中真正联动
4. 让后续测试与 code review 有明确参照

本文件聚焦三条主故事线：

1. 销售故事线
2. 招聘故事线
3. 创始人 / COO 故事线

每条故事线都必须覆盖：
- 联系人
- 公司
- 机会
- 会议
- 邮件线程
- Action items
- Approval tasks
- Audit logs
- Memory facts
- Commitments
- Blockers
- Briefing snapshot
- Recommendation

---

## 一、总原则

### 1. Seed 不是随机数据
Seed 必须像一个真实业务团队最近两周在推进的事情。

### 2. 每条故事线必须可演示
任何一条故事线都应至少支持：
1. 首页看重点
2. 机会面板看推进状态
3. 联系人页看关系和记忆
4. 会议页看 briefing 和会后动作
5. 审批中心批准或拒绝一个动作
6. 记忆页看到事实、承诺、阻碍和修正历史
7. 审计日志看到变化

### 3. 每条故事线都要有“推进”和“阻力”
如果所有事情都顺利，就不真实。
每条故事线至少要有：
1. 一个升温中的机会
2. 一个风险点
3. 一个未完成承诺
4. 一个阻碍
5. 一个需要审批的动作

### 4. 每条故事线都要有“系统记住了什么”
不能只有对象和事件，必须有真正的记忆输出：
- fact
- commitment
- blocker
- snapshot
- recommendation

## 六、跨故事线公共要求

每条故事线都必须满足以下条件：

### 1. 至少有 1 条可审批的高风险动作
例如：
- 对外发送关键邮件
- 对候选人发送关键反馈
- 对合作方承诺下一步

### 2. 至少有 1 条自动执行的低风险动作
例如：
- 内部纪要分发
- 创建内部会议
- 更新时间线

### 3. 至少有 1 条 overdue commitment
因为没有逾期场景，管理者周报和风险提醒都不真实。

### 4. 至少有 1 条 blocker 被解决
否则时间线只会展示“问题存在”，看不到推进和变化。

### 5. 至少有 1 条 memory correction
这是演示“记忆可纠错”的关键。

### 6. 至少有 1 条 recommendation 带 supporting facts
否则看不出记忆系统如何支撑动作建议。

## 十、最后要求

Seed 数据不是辅助材料，它是产品的一部分。

如果 seed 数据讲不出故事：
- 首页就不可信
- 会议页就不打动人
- 审批中心就显得机械
- 记忆系统就像空壳

所以，Codex 在生成 seed 数据时，必须优先追求：
1. 故事性
2. 连续性
3. 可演示性
4. 对象之间关系清晰
5. 变化和后果明确

这比简单追求“数量多”更重要。
