# Pack B — SI / 咨询交付承诺与 Handoff 行业 Pack 调研、定位与需求 V1

## 1. 适用 ICP
AI 服务商 / 系统集成商 / 咨询交付公司（与 ICP playbook ICP 2 对齐）。

**对终端客户：** 把客户承诺、方案边界、交付 Handoff 和异常处理留在可审计链路里，减少销售到交付的断层。
**对自媒体：** "项目承诺—边界—关单"循环的开源样板，与米盾云 worker 家族同构。

## 2. 行业调研
### 2.1 客群画像
| 维度 | 描述 |
|---|---|
| 公司规模 | 30-300 人 SI / 咨询 / AI 服务公司 |
| 业务模式 | 项目制销售 + 定制交付 |
| 项目客单价 | ¥50 万 - ¥1,000 万 |
| 周期 | 1-12 月 |
| 决策买家 | 创始人 / 合伙人 / 交付负责人 / 售前负责人 |
| 工具栈 | CRM（轻度）+ 飞书/钉钉 + 项目管理（PingCode/Jira/Worktile）+ 文档 + 自研工具 |
| AI 现状 | "在帮客户做 AI" 但 "自己用 AI 跟项目" 较少 |

### 2.2 痛点
1. 售前承诺越界——口头答应"能做 X"，交付返工
2. 方案边界模糊——客户不断加需求，没人守
3. 交付 Handoff 断层——售前→项目→CS 接力丢上下文
4. 异常无审计——客户有异议时没有承诺链
5. 项目复盘缺记忆——同样的坑反复踩
6. 合伙人盲区——看不到哪个项目正在出风险

### 2.3 与脱敏行业样板同构
| 脱敏行业样板 | Pack B 映射 |
|---|---|
| Worker D3 PTP 跟进 | 售前/合同承诺跟进 |
| Worker D4 closure | 项目阶段关单 |
| Worker D5 QC 复核 | 客户验收复核 |
| Worker D6 异常 readout | 项目异常 readout |
| observer-first + 不掉案件 invariant | 项目不掉单 invariant |

**核心技术叙事卖点：**"Helm worker 家族：从米盾联通样板到 SI 行业通用 Pack"——可写深度博客。

### 2.4 竞品对比
| 类型 | 代表 | 与 Pack B 差异 |
|---|---|---|
| 项目管理 SaaS | PingCode/Worktile | 止于任务进度；Pack B 是承诺/边界/审计 |
| RPA / 流程自动化 | UIBot / 来也 | 做执行自动化；Pack B 做人在环路判断与审计 |
| 通用 Agent 平台 | 字节扣子 / 百炼 | 是 builder；Pack B 是行业成型方案 |
| 海外标杆 | 无直接对标（SFDC + Slack 拼凑） | Pack B 是中国市场差异化 |

### 2.5 开源传播
**适合，独特技术叙事价值。** SI CTO 强技术评估能力，开源可信度直接驱动决策。圈层比 SaaS 封闭，深度足够但广度不及 Pack A。

## 3. Pack B 定位
### 3.1 主卖点
1. Revenue Operating Loop（项目版）——项目销售推进 + 承诺审计
2. Governed CRM/CS Loop——交付侧合同与现实对账
3. Exception/Reconciliation——项目异常可审计 readout
4. Proof Pack——给客户的可视化交付凭证

### 3.2 No-Go
项目交付高度非标 / 客户只要一次性定制开发 / 没有试点 owner 或周复盘节奏。

## 4. Skill 设计（4 个）
### 4.1 Skill B1：客户承诺审核
- 触发：售前/销售在 IM、邮件、会议产生对外承诺类语句
- 输入：纪要 + 邮件 + IM + 当前方案版本
- 输出：承诺识别清单（口头/书面/隐含）+ 边界冲突 + 复核建议
- 复核点：识别为"承诺"默认待复核；销售/售前确认后入承诺库
- 内核：脱敏 PTP worker 样板改造（D3 模式）

### 4.2 Skill B2：方案边界守护
- 触发：客户提新需求 / 方案变更 / 合同条款变化
- 输入：当前方案 + 历史承诺库 + 预算与工时
- 输出：边界变化清单 + 商业影响测算 + 是否触发合同变更建议
- 复核点：变更默认建议；项目经理或合伙人决策
- 内核：脱敏 closure worker 样板改造（D4 模式）

### 4.3 Skill B3：交付 Handoff Pack
- 触发：售前→项目；项目→交付；交付→CS
- 输入：上游环节的会议、承诺、边界、风险
- 输出：Handoff Pack（承诺/边界/待复核/客户性格画像/风险）
- 复核点：上游 owner 确认后正式提交
- 内核：Handoff Pack worker（与 Pack A A4 共享基础 worker，行业版定制）

### 4.4 Skill B4：项目异常 Readout
- 触发：客户投诉 / 验收失败 / 项目延期超阈值
- 输入：承诺库 + 项目执行日志 + 客户原始诉求
- 输出：异常 readout（时间线/承诺 vs 现实/责任划分/对客户解释草稿）
- 复核点：对客户解释默认建议；合伙人或项目经理审核后正式发出
- 内核：脱敏 QC + 异常 readout 样板（D5+D6 模式）

### 4.5 开源/商业边界
| Skill | 开源 | 商业版（Pack B） |
|---|---|---|
| 通用 worker 家族骨架 | ✅ 与 Pack A 共享 basic Meeting-to-Action / Handoff | — |
| Pack B 行业适配（PTP→承诺；closure→关单；QC→验收） | ❌ | ✅ Pack B certified |
| 米盾联通参考实现 | ❌（自营线，与 Pack B 隔离） | — |
| Cookbook 案例 | ✅ 脱敏样例可开源 | — |

**关键约束：** 米盾联通自营样板间与 Pack B 生态版本严格隔离，避免 Helm 与工程师抢单。

## 6. 数据接入
### 6.1 最小源
| 源 | 用途 | scope |
|---|---|---|
| 飞书/钉钉/企微会议 | 触发 + 纪要 | OAuth + 只读 |
| 邮箱 IMAP | 邮件 | 只读 |
| IM | 沟通历史 | OAuth + 只读 |
| 项目管理 | 任务、进度 | API 只读 + 双向（可选） |
| 文档（飞书/Notion） | 方案版本 | API 只读 |
| 合同/报价（自研） | 条款追踪 | 多样化，需定制 |

### 6.2 接入复杂度
| 系统 | 复杂度 | 工程师 |
|---|---|---|
| 标准飞书/钉钉/企微 | 低 | L1 |
| PingCode/Worktile/Jira | 中 | L2 |
| **自研合同/报价系统** | **高，是 Pack B 核心实施机会** | L2/L3 必须 |
| 多事业部/多公司 | 高 | L3 + Helm 核心联运 |

**Pack B 是认证工程师生态最大受益方**——客户系统多样、定制需求强，每项目 5-15 人天。

### 6.3 私有化
部分客户出于客户数据保密需要私有化；需 Enterprise 合同 + Helm 核心评估；工程师不能擅自交付。

## 7. 不做清单（红线）
| 不做 | 理由 |
|---|---|
| 自动判定"是否承诺" | 识别是建议，判定权在销售本人 |
| 自动发对客户的"异常解释" | 客户关系不能 AI 替代 |
| 自动生成合同变更单 | 法律文件必须人工 |
| 替合伙人评估项目经理 | 不替代管理判断 |
| 自动关单/验收 | 与客户验收冲突 |

**对外措辞红线：** 不得用"自动管理项目"、"AI 替你交付"、"无需项目经理"。

## 8. 试点验收
| 指标 | 4-8 周 pilot | 6 月稳态 |
|---|---:|---:|
| 承诺识别准确率 | ≥75% | ≥90% |
| 越界承诺早期发现率 | ≥50%（客户提出前） | ≥75% |
| 边界变更工时测算误差 | ≤30% | ≤15% |
| Handoff 完整度 | ≥3.5/5 | ≥4.2/5 |
| 异常 readout 生成时间 | 1-2 天 → ≤4h | ≤1h |
| 项目复盘可复用性 | ≥3.0/5 | ≥4.0/5 |

## 10. 风险
1. 米盾自营线与 Pack B 生态线必须严格隔离。**对策**：米盾在金融催收，Pack B 锚定企业 IT 与咨询 SI——行业不重叠
2. 客户系统多样性——合同/报价多自研，工作量难标准化。**对策**：Pack B 提供 SDK 与连接器骨架，工程师做客户化
3. 中文承诺识别准确度——核心技术风险。**对策**：模型微调 + 标注训练集 + 校准回路
4. 首批 design partner ≥2 家——**待补充**
5. L2/L3 工程师 ≥10 名 6 月内是否可达——影响 Pack B 启动速度
