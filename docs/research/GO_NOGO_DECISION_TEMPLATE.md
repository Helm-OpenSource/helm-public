## 📋 如果是Go: MVP范围定义

### 产品范围
**只做一条收入线**: [选择其中一种]
- [ ] SALES_REFERRAL (销售推荐)
- [ ] CUSTOM_IMPLEMENTATION (定制实施)
- [ ] CUSTOM_MAINTENANCE (定制维护)

### 用户界面
**只暴露3个核心概念**:
1. **我的贡献**: 最近贡献事件和状态
2. **我的应计**: 已累计价值和成熟状态
3. **我的结算**: 下批结算时间和预估

**不暴露复杂概念**:
- 不显示: ContributionEvidenceRef, AttributionGraphEdge, AllocationTemplate等
- 不显示: 复杂的评分公式和权重
- 不显示: 争议处理细节（除非用户涉及争议）

### 治理层保留
- [ ] 争议机制: 设计完成，但不做完整产品面
- [ ] 规则版本: 预留扩展接口
- [ ] 证据引用: 预留数据结构
- [ ] 回撤机制: 设计完成，但手动触发

### 不做 (V1明确排除)
- [ ] 自动打款
- [ ] 股权累积
- [ ] 全平台池自动拆分
- [ ] 完整争议解决工作流

### 成功指标 (MVP验证期3个月)
- **行为指标**:
  - 贡献提交率: [目标]%
  - 系统查看频率: [目标]次/人/周
  - 用户满意度: [目标]分

- **业务指标**:
  - GTM团队留存率: [目标]%
  - 外部合作者留存率: [目标]%
  - 争议频率: <5%

- **技术指标**:
  - 系统响应时间: <2秒
  - 数据准确率: >95%

### 风险缓解
- **风险1**: 用户认知负担过高
  - 缓解: 界面只显示3个核心概念
  - 验证: 3个月后的用户调研

- **风险2**: 争议成本过高
  - 缓解: 先做手动争议处理，不做自动化
  - 验证: 统计争议频率和处理成本

- **风险3**: 信息收集成本
  - 缓解: 只在关键节点收集信息，不做全量记录
  - 验证: 监控用户平均信息收集时间
