# 产品路线图

## 文档目的

本文件定义“经营推进控制台”的产品路线图与阶段边界。

作用有四点：

1. 让研发和产品对当前目标保持一致
2. 帮助 Codex 和评审者理解“现在最重要的是什么”
3. 避免版本迭代时目标漂移
4. 明确哪些内容当前不做

本文件只关心接下来 12 个月内可执行的路径，不讨论过远愿景。

---

## 一、产品总方向

经营推进控制台的总方向是：

从高频经营工作流切入，先帮助用户发现和推进下一步动作，再逐步补齐：
1. 工作域记忆
2. 行动执行
3. 授权与审批
4. 审计与可追责
5. 预算与策略控制
6. 多代理协作
7. 信誉、归因与更大的控制层

长期来看，它会从“经营工作台”成长为“可控行动代理的控制层”。

但在当前阶段，所有资源都应集中于：
**高频工作流 + 行动建议 + 审批闭环 + 工作域记忆。**

## 二、路线图总览

### 当前阶段：V0
目标：
把产品做成“可运行、可演示、可试用”的 MVP。

重点：
1. 页面完整
2. 流程闭环
3. Demo 数据可信
4. 审批、机会、会议、时间线联动真实
5. 能支持对外演示和首批客户访谈

### 下一阶段：V1
目标：
把 MVP 做成“可被第一批设计合作伙伴真实使用”的版本。

重点：
1. 更强产品感
2. 更强稳定性
3. 更强策略与审批逻辑
4. 支持真实团队试点
5. 支持更真实的工作域记忆沉淀

### 再下一阶段：V2
目标：
把产品从“可用工具”进化为“可控经营系统”。

重点：
1. 多角色协作
2. 更细化的授权模型
3. 更强动作自动化
4. 更清晰的管理视图
5. 更明显的组织价值

## 三、当前版本目标：V0

### V0 的目标只有五个

1. 用户能打开系统并快速看懂今天该做什么
2. 会议能自动沉淀为行动
3. 机会能被结构化推进
4. 高风险动作能审批
5. 所有关键动作都能审计和回放

### V0 的成功标准

只要满足以下条件，就认为 V0 成立：

1. 首页能清晰展示今日重点
2. 会议详情页可以跑通 briefing 到 action 到审批
3. 机会面板可管理推进状态
4. 联系人和公司详情页有真实工作记忆
5. 审批中心可以真正驱动状态变化
6. 审计日志可追踪关键动作
7. Demo 数据足够支撑销售、招聘、创始人三条演示路径

### V0 必做功能

1. 登录页
2. 初始化页
3. 今日工作台
4. 机会面板
5. 联系人详情页
6. 公司详情页
7. 会议详情页
8. 审批中心
9. 收件箱线程页
10. 记忆与时间线页
11. 设置与策略中心

### V0 必须可讲的三条故事线

1. 销售故事线
2. 招聘故事线
3. 创始人经营故事线

### V0 绝不追求的事情

1. 真实支付
2. 真实第三方集成
3. 复杂流程引擎
4. 多代理编排
5. 企业级后台
6. 大而全报表系统
7. 泛社交或数字人体验

## 四、下一版本目标：V1

### V1 核心目标

让 3 到 6 家设计合作伙伴开始真实试用。

### V1 要解决的问题

1. 从 demo 感过渡到真实产品感
2. 从单用户视角过渡到小团队视角
3. 从“AI 给建议”过渡到“AI 可在规则内帮我做事”
4. 从静态数据过渡到更动态的工作流记录

### V1 重点能力

1. 更强全局搜索
2. 更强通知中心
3. 更清晰的角色与权限
4. 更好的批量操作
5. 更强的审批规则模板
6. 更好的记忆纠错
7. 更完整的会后推进自动化
8. 更可信的 inbox 到机会升级路径
9. 更可讲的工作台和管理视图

### V1 成功标准

1. 设计合作伙伴愿意每周真实使用
2. 用户开始把部分低风险动作交给系统
3. 用户开始依赖首页和审批中心
4. 产品能减少真实漏跟进
5. 客户愿意继续试点或付费扩展

## 七、当前最重要的产品押注

接下来所有迭代，都优先押注以下四个能力：

### 1. 首页价值感
首页必须让用户快速知道：
- 今天什么最重要
- 哪些事情有风险
- 哪些动作在等我
- 系统建议我现在做什么

### 2. 会议到行动闭环
这是最容易打动客户的能力。
每次迭代都要问：
会议详情页是不是更强了？
会后行动是不是更顺了？
审批是不是更可信了？

### 3. 机会推进感
系统不能只是展示机会。
系统必须帮助推进机会。
如果某次迭代没有增强“推进感”，优先级就不够高。

### 4. 审批与审计信任感
产品不是靠“看起来聪明”赢。
产品靠“用户敢逐步放权”赢。
所以每一轮都要强化：
审批体验
策略变化
审计可读性
动作后果

## 八、明确不做清单

下面这些内容在当前阶段明确不做，除非有非常强的商业牵引：

1. 泛聊天主界面
2. 数字人和 3D 形象
3. 情感陪伴
4. 泛社交网络
5. 面向大众消费者的轻应用
6. 真实支付和清算
7. 大而全 CRM 能力
8. 大而全邮件客户端能力
9. 花哨但无业务价值的数据大屏
10. 复杂报表系统
11. 多工作区复杂组织模型
12. 过重的管理员后台
13. 过度通用的工作流编排器

## 九、版本判断原则

每次讨论新需求或新版本，先回答：

1. 这项改动是否增强首页价值感？
2. 这项改动是否增强会议到行动闭环？
3. 这项改动是否增强机会推进感？
4. 这项改动是否增强审批与审计信任感？
5. 这项改动是否服务当前目标用户？
6. 这项改动是否有助于设计合作伙伴试点？

如果其中 4 项以下回答为“是”，这个需求不应优先。

## 十、长期路线图提醒

未来可以延伸到：
1. 更强控制层
2. 更强预算与策略
3. 更强多代理协作
4. 更强信誉与归因
5. 更大的 agent-to-agent 网络

但当前不要直接做这些终局能力。
当前只做一件事：

**把“一个可控的行动代理真的能帮我推进工作”做得足够真。**

这是所有版本的判断基准。

## 十一、continuity surface closeout note

截至 `2026-04-05`，continuity surface 这条线已经收口为 `stable & maintained`。

当前含义是：

1. continuity 相关 readout、operator surface、runbook、eval / e2e、self-check、boundary guard 继续保留
2. continuity 默认进入 `maintenance-only`
3. 只有出现真实风险或回归，continuity 才应重新进入开发

当前不应做的是：

1. 继续沿 continuity 线默认开新的 deepening PR
2. 把 continuity closeout 误读成 execution authority expansion
3. 把 continuity closeout 误读成 auto-remediation、team mode 或 world-model productization 已经获批

如果后续要进入新的大方向，必须另开 scoped plan。
这个 closeout 只负责结束 continuity 的连续深挖开发线，不负责自动批准下一个主方向。
