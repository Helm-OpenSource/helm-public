# Conversation Capture Seed 场景

## 文档目的

本文件用于为“经营分身控制台”的交流捕获系统（Conversation Capture System）定义 seed 场景、demo 场景和人工验收场景。

目标有四个：

1. 让 Codex 在生成 seed 数据时有清晰标准
2. 让交流捕获层的价值可以被直接演示
3. 让人工验收时能验证“现场捕获真的增强了记忆、推荐、审批和管理闭环”
4. 让产品不会退化成一个普通录音或转写工具

---

## 一、总原则

### 1. 场景必须强调“现场感”
所有 seed 场景都要体现：
- 用户在现场开始记录
- 系统在后台处理
- 结果自然回到经营对象和管理闭环

不要把场景设计成“上传音频后做摘要”。

### 2. 场景必须有明确业务结果
每个场景都必须至少带来以下结果中的三项：

1. 生成事实 MemoryFacts
2. 生成 Commitment
3. 生成 Blocker
4. 生成 Risk
5. 生成 Next Best Action
6. 进入审批
7. 更新首页今日重点
8. 更新联系人、公司或机会页

### 3. transcript 只是中间层
seed 里可以有 transcript，但 demo 重点不应停留在 transcript。
重点是 transcript 之后的：
- 经营理解
- 记忆生成
- recommendation
- approval
- management output

### 4. 每个场景都要有“推进”和“阻力”
如果只有平顺信息，没有风险和阻碍，系统价值会很弱。

每个场景至少要有：
1. 一个承诺
2. 一个 blocker
3. 一个需要跟进的动作
4. 一个“如果现在不处理会怎样”的风险

---

## 二、场景总表

建议第一版至少准备 5 个核心场景。

### 场景 A
销售会议现场记录

### 场景 B
招聘面试或面试反馈会话记录

### 场景 C
创始人与外部合作方沟通记录

### 场景 D
内部优先级协调会记录

### 场景 E
Call Center / 电话场景外部接入

补充说明：
- 当前代码已经支持通过 ingest API 提交外部 transcript
- 第一版重点不是接通 Zoom / 腾讯会议 SDK，而是先把“外部 transcript -> insight -> memory / recommendation / approval”打成统一闭环

## 三、场景 A：销售会议现场记录

## 场景目标
证明“开始记录”后，系统能把客户会议自动转成经营推进闭环。

## 角色
- 用户：saleslead@demo.com
- 客户联系人：李晨，采购负责人
- 客户业务联系人：周颖，业务总监
- 公司：星桥科技
- 机会：星桥科技年度方案合作

## 会话背景
当前机会正在升温，但还存在两个关键问题：
1. 客户真正更关心付款节奏，不是总价
2. 下周三前希望看到精简版方案结构

内部还有一个问题：
- 产品团队资源尚未确认

## 前台动作
用户在“星桥科技年度方案合作”机会页点击“开始记录”。

## 录音后的 transcript 应体现的内容
建议 transcript 里包含以下语义点：

1. 客户希望下周三前收到东西
2. 客户更关心付款周期和推进节奏
3. 周颖整体支持推进
4. 李晨还在等内部预算 / 财务口径确认
5. 你方内部还要确认交付节奏

## 系统应自动生成的结果

### MemoryFacts
至少包括：
1. 客户更关心付款周期
2. 周颖支持项目推进
3. 李晨在等内部预算 / 财务口径确认
4. 当前推进重点是先给精简版方案结构

### Commitments
至少包括：
1. 下周三前发送精简版方案结构草稿
2. 内部两天内确认交付节奏

### Blockers
至少包括：
1. 客户内部预算 / 财务口径尚未完全确认
2. 内部交付节奏尚未拍板

### Risks
至少包括：
1. 如果 24 小时内不推进，客户期待会下降
2. 如果先讨论价格而不是付款节奏，推进可能继续卡住

### Next Best Actions
至少包括：
1. 发精简版方案结构草稿
2. 内部确认交付节奏
3. 对外动作进入审批

## 页面体现

### 首页
应出现：
1. 星桥科技机会进入“今日最值得推进的三件事”
2. 出现一个待审批动作
3. 出现一个高风险 blocker 提示

### 会议页
应出现：
1. 本次会议生成的 facts
2. 本次会议生成的 commitments
3. 本次会议生成的 blockers
4. next best action
5. recommendation explanation

### 联系人页
李晨页应出现：
1. 最近一次会话摘要
2. 当前最关键关切
3. 当前未完成承诺
4. 推荐下一步动作

### 机会页
应出现：
1. 当前最大 blocker
2. 当前 commitments
3. recommendation explanation 引用了这次会话结果

### 审批中心
应出现：
1. 发精简版方案结构草稿待审批
2. explanation 引用了本次会话里的关键事实

## 验收标准
1. 用户结束记录后，系统能自动生成 transcript
2. transcript 能转成结构化经营结果
3. 结果能写回首页、会议页、联系人页、机会页、审批中心
4. recommendation 能引用这些结果

## 四、场景 B：招聘面试 / 反馈会话记录

## 场景目标
证明系统能把面试和招聘沟通自动转成候选人推进闭环。

## 角色
- 用户：recruiter@demo.com
- 候选人：刘然
- 客户联系人：许薇，HRBP
- 公司：悦途医疗
- 机会：高级算法工程师岗位

## 会话背景
刘然已经完成技术面，客户希望尽快推进，但存在两个问题：
1. 候选人对薪资有顾虑
2. 客户反馈还未同步，follow-up 快逾期

## 前台动作
用户在“刘然”联系人页或“高级算法工程师岗位”机会页点击“开始记录”。

## transcript 应体现的内容
建议包括：
1. 刘然认可岗位方向
2. 刘然对薪资区间仍有顾虑
3. 客户倾向推进到下一轮
4. 希望 48 小时内同步反馈

## 系统应自动生成的结果

### MemoryFacts
1. 刘然认可岗位方向
2. 刘然对薪资区间有顾虑
3. 客户倾向继续推进
4. 反馈同步时效要求高

### Commitments
1. 48 小时内同步面试反馈
2. 本周内安排下一轮沟通

### Blockers
1. 薪资 gap
2. 反馈延迟风险

### Risks
1. 超过 48 小时不反馈，候选人流失风险上升

### Next Best Actions
1. 先同步积极反馈
2. 单独讨论薪资边界
3. 安排下一轮沟通

## 页面体现
### 联系人页
应前置：
1. 当前顾虑
2. 当前 commitment
3. 当前 blocker
4. 推荐下一步动作

### 机会页
应看到：
1. 招聘机会的当前 blocker
2. commitment
3. next best action

### 审批中心
如果生成对外反馈草稿，应进入审批。

### 周报
应体现：
1. 招聘流程中的 follow-up 风险
2. 候选人流失高风险提醒

## 验收标准
1. 会话结束后，系统能识别候选人真实顾虑
2. recommendation 不会只说“继续推进”，而是会考虑薪资和反馈时机
3. 结果能进入联系人页、机会页和审批中心

## 五、场景 C：创始人与外部合作方沟通记录

## 场景目标
证明系统能理解合作场景下的复杂承诺、阻碍和内部资源冲突。

## 角色
- 用户：founder@demo.com
- 合作方联系人：陈嘉
- 公司：Northlight AI
- 机会：Northlight AI 渠道合作

## 会话背景
合作方积极，但内部资源冲突，创始人需要在短时间内决定是否给出轻量方案路径。

## 前台动作
用户在合作机会页点击“开始记录”。

## transcript 应体现的内容
建议包括：
1. 对方合作意愿强
2. 两天内希望收到下一步方案
3. 内部产品负责人担心排期受影响
4. 轻量方案可能是可行路径

## 系统应自动生成的结果

### MemoryFacts
1. 合作方积极
2. 内部资源紧张
3. 轻量合作路径可行

### Commitments
1. 两天内给出下一步方案

### Blockers
1. 内部资源冲突
2. 优先级尚未统一

### Risks
1. 若两天内不回复，合作势能可能下降

### Next Best Actions
1. 先确认内部轻量方案边界
2. 向合作方发出阶段性回复
3. 对外动作进入审批

## 页面体现
### 首页
应出现：
1. 该合作机会出现在今日重点里
2. 内部优先级冲突以高风险事项出现

### 公司页
Northlight AI 页应看到：
1. 当前合作势能
2. blocker
3. commitment
4. 推荐推进路径

### 机会页
应出现：
1. 当前 blocker
2. 下一步推荐
3. 若不推进的影响

### 审批中心
应看到：
1. 对外阶段性回复待审批

## 验收标准
1. 系统能识别“对方积极 + 内部资源紧张”的双重现实
2. recommendation 是先解决资源冲突，再对外回复
3. 页面体现真实经营判断

---

## 六、场景 D：内部优先级协调会记录

## 场景目标
证明系统不只适用于外部沟通，也适用于内部经营管理和优先级决策。

## 角色
- 用户：founder@demo.com
- 内部产品负责人：宋逸
- 内部事项：优先级协调

## 会话背景
多个项目争抢资源，需要快速决定谁优先。

## 前台动作
用户在某内部事项或会议页点击“开始记录”。

## transcript 应体现的内容
建议包括：
1. 某外部合作推进快，但内部资源不够
2. 另一个已签项目有交付压力
3. 若不尽快决策，会影响外部承诺兑现

## 系统应自动生成的结果

### MemoryFacts
1. 当前资源冲突真实存在
2. 外部推进速度快于内部资源准备速度
3. 某已签项目交付不可轻易延后

### Commitments
1. 本周内完成资源优先级决策

### Blockers
1. 内部优先级冲突
2. 负责人尚未统一意见

### Risks
1. 外部承诺可能逾期
2. 内部交付压力加大

### Next Best Actions
1. 创始人拍板优先级
2. 给外部合作方一个过渡性回复
3. 更新内部排期动作

## 页面体现
### 首页
应出现：
1. 内部事项作为今日重点
2. 外部承诺逾期风险提醒

### 周报
应体现：
1. 资源冲突类 blocker 增加
2. 承诺兑现压力上升

## 验收标准
1. 系统能把内部会也转成经营风险和行动
2. 产品不会只服务外部沟通场景

## 七、场景 E：Call Center / 电话系统外部接入

## 场景目标
证明产品不仅能手动现场记录，也支持企业级外部系统接入。

## 角色
- 来源：销售 Call Center 或电话系统
- 用户：saleslead@demo.com
- 场景：电话跟进客户

## 输入方式
通过 API / CLI ingest，不对普通用户开放上传入口。

建议输入：
1. 基础元数据
2. 参与方
3. transcript
4. 关联对象

## transcript 应体现的内容
建议包括：
1. 客户语气积极但拖延
2. 某个 blocker 被再次确认
3. 需要尽快发后续资料

## 系统应自动生成的结果
1. facts
2. commitments
3. blockers
4. recommendation
5. approval，若动作高风险

## 页面体现
### 收件箱 / 联系人 / 机会页
应能看出：
1. 本次会话来自外部系统
2. 系统已处理出经营信号
3. recommendation explanation 能引用该会话

## 验收标准
1. ingest 链路正常
2. 不需要普通用户手工上传录音
3. 外部系统会话也能回到经营闭环

---

## 八、必须覆盖的 UI 感受

这些 seed 场景必须让用户感受到四件事：

1. 前台非常简单  
   用户只做“开始记录”和“结束记录”

2. AI 在后台处理很重  
   转写、提取、记忆生成、推荐都自动发生

3. 输出直接回到经营系统  
   不是停在 transcript

4. 结果会继续推动事情  
   recommendation、审批、action item、首页都发生变化

---

## 九、seed 数量建议

第一版建议最少覆盖：

### CaptureSession
- 5 到 8 个

### ConversationTranscript
- 5 到 8 个

### ConversationInsight
至少 25 到 40 条，覆盖：
- FACT
- COMMITMENT
- BLOCKER
- RISK
- NEXT_ACTION

### 相关输出
- 至少 5 组 recommendation explanation 引用 conversation insights
- 至少 3 条高风险动作进入审批
- 至少 3 条首页高优先事项来自会话捕获

---

## 十、最小演示路径

建议固定 4 条演示路径。

### 路径 1：销售会议现场记录
开始记录 → 结束 → transcript → commitments / blockers → recommendation → 审批

### 路径 2：招聘面试反馈记录
开始记录 → transcript → 候选人顾虑识别 → follow-up recommendation

### 路径 3：创始人合作沟通记录
开始记录 → 资源冲突识别 → 对外回复进入审批

### 路径 4：外部系统 ingest
通过 API 接入会话 → 自动生成经营对象更新

---

## 十一、人工验收关键问题

每次验收都要回答：

1. 没有这条会话捕获链时，这个场景能否成立
2. 加上它以后，系统到底变强在哪里
3. recommendation 是否真的更可信
4. 记忆是否更真实
5. 用户是否会更依赖这个入口
6. 是否仍然足够简洁和可控

---

## 十二、最后提醒

Conversation Capture 的 seed 场景，不能只证明“我们会转写”。

它必须证明：

1. 系统真的进入了现场
2. 系统真的把对话转成了经营记忆
3. 系统真的把这些记忆转成了 recommendation 和管理闭环
4. 用户会因此更依赖这个产品，而不是把它当成录音工具
