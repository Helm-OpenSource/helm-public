# 主动进化系统设计

## 文档目的

本文件用于定义“经营分身控制台”的主动进化系统（Adaptive Evolution System）。

目标有五个：

1. 让系统能够基于增量变化持续调整推荐、优先级和默认行为
2. 让这种进化建立在真实业务信号之上，而不是黑盒漂移
3. 让用户能感知到“系统越来越懂我和我的团队”
4. 让系统的进化始终处于可解释、可确认、可撤销的控制边界内
5. 让这套能力成为产品区别于传统 SaaS / CRM 的关键层

---

## 一、什么叫“主动进化”

这里的“主动进化”，不是技能越来越多，也不是模型版本升级。

这里的主动进化，是指系统能基于每天新增的变化，持续更新下面四类内容：

1. recommendation 的排序权重
2. 风险识别与优先级阈值
3. 用户与团队的偏好信号
4. 策略建议和默认处理方式

它的基础不是“更强大模型”，而是“更强的增量变化理解”。

---

## 二、为什么这一层必要

目前产品已经有三层能力：

1. 记忆  
   系统知道发生了什么

2. 判断  
   系统会给 recommendation、briefing 和 next best action

3. 执行  
   系统能在规则内推动一些动作发生

但高粘度产品还缺第四层：

4. 进化  
   系统会根据新增变化，调整自己的工作方式

如果没有第四层，系统永远只是：
“当前这一刻会建议”

而很难变成：
“越用越像你的分身”

---

## 三、主动进化系统的核心问题

主动进化系统要回答四个问题：

1. 什么变化值得被系统记录为“模式信号”
2. 这些模式信号会影响系统的哪一层
3. 系统如何更新自己的默认判断
4. 用户如何理解、确认或撤销这种进化

---

## 四、系统边界

主动进化系统可以做：

1. 更新 PreferenceSignal
2. 调整 recommendation 排序权重
3. 建议策略变化
4. 更新对象风险感知
5. 生成“系统最近学到了什么”的提示

主动进化系统不能直接做：

1. 自动改变高风险审批规则
2. 自动执行原本不允许自动执行的动作
3. 绕过用户授权改变默认外发逻辑
4. 在用户不知情的情况下修改关键业务状态

换句话说：

**系统可以学习，但不能偷偷改规则。**

---

## 五、主动进化系统的五层结构

## 第一层：Delta Event 层
作用：
把系统中的增量变化统一抽象成可分析事件。

典型 Delta Event：
1. recommendation 被批准
2. recommendation 被拒绝
3. recommendation 被编辑后批准
4. 某 blocker 新增
5. 某 commitment 逾期
6. 某 commitment 被兑现
7. 用户修正了一条关键记忆
8. 用户修改策略
9. 某联系人长时间未互动
10. opportunity 阶段变化

这一层的目标是：
把“发生了什么变化”标准化。

## 第二层：Pattern Detection 层
作用：
从 Delta Event 中识别“模式”，而不是只看单次事件。

典型模式：
1. 连续 5 次外发消息都被编辑后批准
2. 某类 blocker 连续两周都是 budget
3. 某团队对会后 24 小时内 follow-up 的批准率显著更高
4. 某用户几乎总是拒绝自动外发
5. 某类机会超过 5 天未推进时掉温概率很高

这一层的目标是：
区分一次性异常和稳定模式。

## 第三层：Adaptive Signals 层
作用：
把识别出的模式沉淀为系统可消费的信号。

建议沉淀三类信号：

1. PreferenceSignal  
   用户和团队的偏好

2. PatternFact  
   对业务规律的观察

3. StrategySuggestion  
   建议用户调整的策略

## 第四层：Adaptive Policy / Ranking Layer
作用：
把这些信号反馈到系统行为里。

主要影响：
1. recommendation ranking
2. today focus 排序
3. risk scoring
4. policy recommendation
5. explanation wording

## 第五层：Evolution UI 层
作用：
把系统最近“学到的东西”显性展示给用户和管理者。

典型展示位置：
1. 首页
2. 审批中心
3. 设置与策略页
4. 周报
5. recommendation explanation

---

## 六、哪些变化值得进入“进化系统”

不是所有变化都值得学习。

我建议按三类处理。

### A. 强信号变化
可以进入长期模式层。

例如：
1. 连续 5 次编辑后批准某类 recommendation
2. 某类 blocker 连续高频出现
3. 某类 recommendation 批准率显著高
4. 某类 opportunity 的掉温模式稳定出现

### B. 中信号变化
先进入观察状态。

例如：
1. 某个用户最近 2 次拒绝某类动作
2. 某个 blocker 本周明显增多
3. 某个联系人的关系温度下降

### C. 弱信号变化
只保留为事件，不进入模式层。

例如：
1. 一次临时性拒绝
2. 一次异常会议改期
3. 一次偶发性延迟回复

---

## 七、系统要进化的四个方向

## 1. 个体偏好进化
系统逐渐学会：
1. 这个用户喜欢什么样的草稿
2. 哪些动作一定要审批
3. 哪些动作可以自动执行
4. 什么时间做 follow-up 更合适

## 2. 团队习惯进化
系统逐渐学会：
1. 这个团队的推进节奏
2. 哪些事项在这个团队里最容易拖
3. 哪些 blocker 是高频规律
4. 哪些 recommendation 在这个团队里最有效

## 3. 对象规律进化
系统逐渐学会：
1. 某类联系人在意什么
2. 某类客户的典型阻碍
3. 某类机会的掉温规律
4. 某类候选人的流失信号

## 4. 策略建议进化
系统逐渐能提出：
1. 建议提升某类事项优先级
2. 建议把某类动作改为逐条审批
3. 建议把某类提醒阈值提前
4. 建议调整某类 follow-up 窗口

---

## 八、主动进化的可见化方式

如果用户感知不到系统在进化，这层价值就不成立。

建议至少做四种可见化。

### 1. 首页“系统最近学到了什么”
例如：
1. 你过去一周通常会先审批对外承诺类动作
2. 你团队对会后 24 小时内的跟进动作采纳率最高
3. 招聘场景中，薪资顾虑已成为当前最高频 blocker

### 2. recommendation explanation 中显式引用学习结果
例如：
1. 基于你过去常把外发消息改短，系统优先给出精简草稿
2. 基于你团队最近对预算 blocker 的处理节奏，系统提高了相关事项优先级

### 3. 策略中心里的“建议调整”
例如：
1. 建议把外发承诺类动作默认设为逐条审批
2. 建议把 follow-up 默认窗口从 72 小时收紧为 48 小时

### 4. 周报里的“规律观察”
例如：
1. 本周 stalled opportunities 中，budget blocker 占比明显上升
2. 本周外发消息类 recommendation 编辑后批准率明显升高

---

## 九、与现有系统的关系

主动进化系统不是替代现有系统，而是站在它们之上。

### 与记忆系统的关系
主动进化系统用记忆系统提供的：
- MemoryFact
- Commitment
- Blocker
- MemoryCorrection

来理解哪些变化值得学习。

### 与 recommendation engine 的关系
主动进化系统不是直接给 recommendation。
它更多是影响 recommendation 的：
- personalization
- urgency
- risk
- policyFit
- explanation wording

### 与策略系统的关系
主动进化系统不直接改策略。
它生成 StrategySuggestion，等待用户确认。

### 与审计系统的关系
主动进化系统的每次模式变化、建议变化、用户确认，都必须可审计。

---

## 十、最终判断

这套系统最终应该让用户感觉：

1. 系统不只是记录我的工作
2. 系统不只是给我建议
3. 系统会根据我的真实变化不断调优自己
4. 但这种调优始终在我的控制范围内

这才是“经营分身”与普通 AI SaaS 的真正差别。
